JavaScript

Bring Adminterface to life with JavaScript. Learn about plugins, our data and programmatic API options, and more.

tip

We've rewritten ActiveAdmin's JavaScript and made it more flexible. In addition, jQuery is no longer a dependency.

Quick Start#

You can export commonly used components so that you can attach them programmatically.

import * as Adminterface from '@cmdbrew/adminterface'

Debugging#

When the DOM loads, the app attaches JavasScript for each component to the window.adminterface object to give you a whole picture of the entire application and allow quick debugging.

Inside your browser JavaScript console
// Enables the debugger> adminterface.debug = true// List available commands and properties> adminterface.help()

Initializers#

Adminterface comes with initializers that you can hook into your custom JavaScript when manipulating DOM dynamically.

NameParamsDescription
initFlatpickrelementFind and initialize flatpickr for child elements with the [data-aa-flatpickr] attribute within the element.
initTomSelectelementFind and initialize TomSelect for child elements with the [data-aa-tom-select] attribute within the element.
initInputCounterelementFind and initialize Input Counter for child elements with the [data-aa-input-counter] attribute within the element.
initPasswordVisibilityelementFind and initialize PasswordVisibility for child elements with the [data-aa-password-visibility] attribute within the element.
initBootstrapelementFind and initialize Bootstrap's JavaScript for child elements matched the specified selectors within the element.
initDetachedDropdownelementFind and initialize Detached Dropdown for child elements with the .dropdown class within the element.

Template#

All initializers follow a similar format. You can also add custom initializers as you see fit.

For example, here is how we initialize flatpickr inside Adminterface:

/* global adminterface */import flatpickr from 'flatpickr'
const initFlatpickr = (element) => {  const flatpickrTriggerList = [].slice.call(element.querySelectorAll('[data-aa-flatpickr]'))
  flatpickrTriggerList.map((el) => {    const options = JSON.parse(el.dataset.aaFlatpickr || '{}')    const instance = flatpickr(el, options)
    adminterface.addObserver(el, instance, 'flatpickr')    return instance  })}
document.addEventListener('DOMContentLoaded', () => {  initFlatpickr(document)})
document.addEventListener('confirm_dialog:before_open', (el) => {  initFlatpickr(el.detail.dialogForm)})
export default initFlatpickr

Usage#

You can call these initializers on a specific event or dynamically inside your script.

import {  initBootstrap,  initDetachedDropdown,  initFlatpickr,  initInputCounter,  initPasswordVisibility,  initTomSelect} from '@cmdbrew/adminterface'
// Call initializers inside a functionconst myFunction = (element) => {  initBootstrap(element)  initDetachedDropdown(element)  initFlatpickr(element)  initInputCounter(element)  initPasswordVisibility(element)  initTomSelect(element)}
// Call initializers on a specific eventdocument.addEventListener('customEvent', () => {  const el = document.getElementById("my-element")
  initBootstrap(el)  initDetachedDropdown(el)  initFlatpickr(el)  initInputCounter(el)  initPasswordVisibility(el)  initTomSelect(el)})

Confirm Dialog#

Confirm dialogs are popups that ask the user for confirmation when performing a specific action. Adminterface uses ConfirmDialog for batch action and delete confirmations. See the guides on BatchActions and ConfirmDialog for more information.

Configuration#

You can override the default configurations by following the instructions below:

NameDescription
confirmThe confirm button
cancelThe cancel button
  • CSS Class
  • Translation
To update the default CSS classes, you can create a css_claases.yml and specify the changes. See Configuration for more information.

e.g., config/adminterface/admin/css_classes.yml
---confirm_dialog:  cancel: btn btn-light  confirm: btn btn-primary
To update the translations, you can create an active_admin.en.yml file and specify the keys you want to update.

config/locales/active_admin.en.yml
---en:  active_admin:    confirm_dialog:      cancel: Cancel      confirm: Okay!

Events#

We've exposed a few events for hooking into the modal dialog's functionality.

TypeDescription
confirm_dialog:before_openThis event fires before the dialog is shown
confirm_dialog:after_openThis event fires after the dialog is shown
confirm_dialog:confirmThis event fires after the user clicks the confirm button
confirm_dialog:cancelThis event fires after the user clicks the cancel button
document.addEventListener('confirm_dialog:before_open', (el) => {  // Add your custom code here...  console.log(el.detail.dialogForm)})

Usage#

You can specify the configurations by passing in the data option inside your HTML.

  link_to(    "Confirm", "fake/path",    data: {      confirm: "My confirm message",      "aa-confirm-dialog": {        buttons: {          confirm: {text: "Okay!"}        }      }    }  )

You can also trigger the dialog programmatically. Here is an example for adding a confirm dialog to a specific page:

import { ConfirmDialog } from '@cmdbrew/adminterface'
document.addEventListener('DOMContentLoaded', () => {  const targetClassList = ['admin_users', 'show']  const bodyClassList = document.body.classList
  if (targetClassList.every((v) => bodyClassList.contains(v))) {    const message = 'Tell me about yourself!'    const inputs = {      name: 'string',      birth_date: 'date_picker'    }    const options = {      buttons: {        confirm: {          text: 'Okay',          class: 'btn btn-success'        },        cancel: {          text: 'Cancel',          class: 'btn btn-link'        }      }    }    const onConfirm = () => alert('confirmed!')
    new ConfirmDialog(message, inputs, options, onConfirm)  }})

Detached Dropdown#

Adminterface uses DetachedDropdown inside the #active_admin_content element to detach the dropdown menu and append it to the body to prevent overflow cutoffs from scrollable parent elements.

Configuration#

NameDescription
menuThe selector for the element that should be append to body when the dropdown opens

Usage#

Here is an example of adding a detached dropdown to an element:

  • JavaScript
  • HTML
import { DetachedDropdown } from '@cmdbrew/adminterface'
document.addEventListener('DOMContentLoaded', () => {  const dropdownTriggerList = [].slice.call(element.querySelectorAll('.my-dropdown'))  const options = {    menu: '.my-dropdown-menu'  }
  dropdownTriggerList.map((el) => new DetachedDropdown(el, options))})
<div class="my-dropdown dropdown">  <a class="btn btn-secondary dropdown-toggle" href="#" role="button" id="dropdownMenuLink" data-bs-toggle="dropdown" aria-expanded="false">    Dropdown link  </a>
  <ul class="my-dropdown-menu dropdown-menu" aria-labelledby="dropdownMenuLink">    <li><a class="dropdown-item" href="#">Action</a></li>    <li><a class="dropdown-item" href="#">Another action</a></li>    <li><a class="dropdown-item" href="#">Something else here</a></li>  </ul></div>

Usage#

Return HTML based on an input type#

Here is how you can use the Input class to return the HTML of an input type.

// Import the Input classimport { Input } from '@cmdbrew/adminterface'
// Render inputconst html = (new Input("name", {as: "string"})).render()
// Log to consoleconsole.log(html)

Change an input type to a custom class#

Here is an example for changing an input type to use a custom input:

// Import the input you want to extend fromimport { StringInput } from '@cmdbrew/adminterface'
// Define your custom inputclass CustomStringInput extends StringInput {  _defaultInputHTMLOptions () {    const options = {      class: `form-control custom-class ${this.options.as}`,      type: 'string'    }
    return { ...super._defaultInputHTMLOptions(), ...options }  }
  render () {    return this._inputWrapping(`      ${this._labelHTML()}      ${this._inputHTML()}      <p>My Custom Input!</p>    `)  }}
// Update the mappingadminterface.meta.inputs.string = CustomStringInput

Add a custom input type#

Here is an example for adding a custom input type:

// Import the input you want to extend fromimport { BaseInput } from '@cmdbrew/adminterface'
// Define your custom inputclass MyCustomInput extends BaseInput {  _defaultInputHTMLOptions () {    const options = {      class: `form-control custom-class ${this.options.as}`,      type: 'string'    }
    return { ...super._defaultInputHTMLOptions(), ...options }  }
  render () {    return this._inputWrapping(`      ${this._labelHTML()}      ${this._inputHTML()}      <p>My Custom Input!</p>    `)  }}
// Add the custom type to the mappingadminterface.meta.inputs.my_custom_input = MyCustomInput

You can now use this custom input type inside your batch action form. You can learn more about the Batch Action form in the Index Page doc.

ActiveAdmin.register User do  batch_action :custom, form: {    reason: :my_custom_input,  } do |ids, inputs|    # ...  endend

Add a custom plugin for an input type#

Here we will show you an example of how you can add a custom plugin to an input type.

First, add your custom component to a configurations file:

config/adminterface/admin/users/components.yml
...inputs:  string:    js:      - name: my-custom-plugin        default: false        options:          myOption: "test"

Next, load the components configuration file inside your resource and add an input:

app/admin/users.rb
ActiveAdmin.register User do  config.components = YAML.load_file(Rails.root.join("config/adminterface/admin/users/components.yml").to_s)
  form do |f|    f.input :name, as: :string, my_custom_plugin: true  endend

You can see that data-aa-my-custom-plugin="{"myOption":"test"}" has been added into the input element if you inspect the element from the browser console.

Finally, you can attach custom JavaScript for this plugin:

class MyCustomPlugin {  constructor (element, options) {    // Add your custom code...  }}
// Find and iniatilzie the plugin for elements that match the selectorconst initMyCustomPlugin = function (element) {  const myCustomPluginTriggerList = [].slice.call(element.querySelectorAll('[data-aa-my-custom-plugin]'))
  myCustomPluginTriggerList.map((el) => {    const options = JSON.parse(el.dataset.aaMyCustomPlugin || '{}')
    return new MyCustomPlugin(el, options)  })}
// Load the plugin on DOM readydocument.addEventListener('DOMContentLoaded', () => {  initMyCustomPlugin(document)})
// Load the plugin on the `confirm_dialog:before_open` eventdocument.addEventListener('confirm_dialog:before_open', (el) => {  initMyCustomPlugin(el.detail.dialogForm)})
